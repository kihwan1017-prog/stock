## 용어 설명

| 쉬운 말 | 전문 용어 | 뜻 |
|---------|-----------|-----|
| 건강 점검 | `/health` | 서버·DB·AI 상태 한눈에 보기 |
| 꺼짐/장애 | DOWN | 해당 부품이 응답하지 않음 |
| 비상 정지 | Kill Switch | 주문을 막는 스위치 |

| 스케줄러 | Scheduler | 정해진 시간 자동 작업 |

## 1. 먼저 확인할 5가지

1. 브라우저에서 http://127.0.0.1:8000/health 열기  
2. Windows에서 **PostgreSQL 서비스**가 실행 중인지  
3. **Ollama** 가 켜져 있는지 (AI를 쓸 때)  
4. 실전 주문이 실수로 켜져 있지 않은지 (`KIWOOM_LIVE_ORDER_ENABLED`)  
5. 비상 정지 상태: `GET /api/v1/risk/kill-switch`

**[스크린샷]** `/health` 화면 (어느 항목이 DOWN인지 표시)

하나라도 이상하면 아래 “증상별 해결”로 이동합니다.

---

## 2. 증상별 해결

### 2-1. 데이터베이스에 연결이 안 됨

| 보이는 증상 | 할 일 |
|-------------|--------|
| `/health` 에서 database DOWN | PostgreSQL 서비스 시작, `DB_*` 설정 확인 |
| 서버가 아예 안 켜짐 | `scripts\test_db_connection.py` 실행 |
| Alembic 오류·머리가 둘 | `alembic heads` 확인 후 관리자에게 문의 |

**[스크린샷]** DB 서비스가 실행 중인 Windows 서비스 화면

### 2-2. 키움 로그인·연동이 안 됨

| 보이는 증상 | 할 일 |
|-------------|--------|
| 토큰 실패 | `/docs` 에서 `kiwoom/token/test`, 앱키·시크릿 확인 |
| 연습만 됨 | `KIWOOM_USE_MOCK` 값 확인 |
| 실전 주문이 거절됨 | `KIWOOM_LIVE_ORDER_ENABLED`, 실전 전환 승인 여부 |
| 계좌가 안 맞음 | 설정 계좌번호와 스냅샷 계좌번호 형식 비교 |

### 2-3. 업비트가 안 됨

| 보이는 증상 | 할 일 |
|-------------|--------|
| 시세·동기화 실패 | 네트워크, `UPBIT_BASE_URL`, `GET /api/v1/upbit/markets` |
| 실시간 끊김 | `/api/v1/realtime-quotes/status` 확인 후 재시작 API |

### 2-4. AI / Ollama 오류

| 보이는 증상 | 할 일 |
|-------------|--------|
| `/health` ollama DOWN | Ollama 앱 실행, `OLLAMA_BASE_URL` 확인 |
| AI API 오류 | 모델 이름(`OLLAMA_MODEL`), Ollama 쪽 로그 |
| 관리 화면 AI 메뉴 없음 | 준비 중 — `/docs` API로 확인 |

**[스크린샷]** Ollama 실행 + health 정상

### 2-5. 스케줄러가 멈춤

| 보이는 증상 | 할 일 |
|-------------|--------|
| 자동 작업이 안 돌아감 | `SCHEDULER_ENABLED`, `run_scheduler.py` 창이 열려 있는지 |
| Windows 작업이 없음 | `install_scheduler_task.ps1` |
| 한 번만 다시 돌리기 | `run_scheduler_job.py` 또는 scheduler-admin API |

### 2-6. 관리 API가 401 (거절)

- 설정의 `ADMIN_API_KEY` 와 요청 헤더 `X-Admin-API-Key` 가 같은지  
- 키가 비어 있으면 이 PC에서는 통과, 운영 환경에서는 키 필수

### 2-7. 주문이 나가지 않음

순서대로 확인:

1. Kill Switch(비상 정지)가 켜져 있지 않은지  
2. Outbox(대기함)에 PENDING/FAILED 가 쌓여 있지 않은지  
3. 실전 모드라면 실전 전환이 승인됐는지  
4. `KIWOOM_LIVE_ORDER_ENABLED=false` 이면 **실전 주문이 막히는 것이 정상**

### 2-8. 알림이 안 옴

- Telegram / Slack / Discord 설정·열쇠가 맞는지  
- 같은 알림을 너무 자주 보내 건너뛴(SKIPPED) 것은 아닌지

## 3. 기록(로그) 어디서 보나

1. **서버·스케줄러 터미널** (가장 중요)  
2. `GET /api/v1/audit/events` (관리 키)  
3. `GET /api/v1/system/dashboard`  
4. 작업 이력: `GET /api/v1/jobs/history`

**[스크린샷]** 터미널에 빨간 에러가 보인 예 (비밀번호는 가리기)

## 4. 상황별 복구 요약

| 상황 | 복구 순서 |
|------|-----------|
| 서버가 죽음 | 서버 다시 켜기 → `/health` |
| DB가 깨짐 | [백업복구매뉴얼.md](백업복구매뉴얼.md) |
| 실전 주문이 이상함 | Kill Switch ON → 실전 전환 해제 → 원인 조사 |
| 스케줄러 정지 | 다시 켜기 → 필요 시 작업 한 번 수동 실행 |
| 코드를 예전으로 | `git` 되돌리기 + 필요 시 DB도 맞추기 (신중) |
